</span><span class="stx-line" id="code_14">Abstract
</span><span class="stx-line" id="code_15">
</span><span class="stx-line" id="code_16">   This document defines a protocol named WiSH which supports bi-
</span><span class="stx-line" id="code_17">   directional message-based communication over the standard HTTP
</span><span class="stx-line" id="code_18">   semantics.  The WiSH protocol layers the WebSocket (compatible)
</span><span class="stx-line" id="code_19">   framing over HTTP.  You may want to think about WiSH as a binary and
</span><span class="stx-line" id="code_20">   bi-directional alternative to the framing defined for server-sent
</span><span class="stx-line" id="code_21">   events [SSE].

</span><span class="stx-line" id="code_76">1.  Introduction
</span><span class="stx-line" id="code_77">
</span><span class="stx-line" id="code_78">   The WebSocket was proposed to provide client-server bi-directional
</span><span class="stx-line" id="code_79">   messaging for the Web natively.  It's been implemented and deployed
</span><span class="stx-line" id="code_80">   widely, but there're still missing semantics and functionalities.
</span><span class="stx-line" id="code_81">   See [BidiwebSurvey].
</span><span class="stx-line" id="code_82">
</span><span class="stx-line" id="code_83">   The WiSH protocol is a protocol for use over the standard HTTP
</span><span class="stx-line" id="code_84">   semantics to provide bi-directional messaging semantics.  The WiSH
</span><span class="stx-line" id="code_85">   protocol layers a simplified WebSocket framing over the HTTP body.
</span><span class="stx-line" id="code_86">   WiSH stands for Web in Strict HTTP.  The lower layer protocol
</span><span class="stx-line" id="code_87">   providing the HTTP semantics can be HTTP/1.1 [RFC7231], HTTP/2
</span><span class="stx-line" id="code_88">   [RFC7540], HTTP/2 + QUIC [QUIC] , or any future protocols.
</span><span class="stx-line" id="code_89">   Transport-level functionalities such as compression, multiplexing,
</span><span class="stx-line" id="code_90">   session priority, etc. are provided by the lower layer wire protocol
</span><span class="stx-line" id="code_91">   [TransportAbstraction].  Unlike HTTP/2, HTTP/1.1 doesn't specify if
</span><span class="stx-line" id="code_92">   earlier 2xx responses are allowed [RFC7540].  Therefore when HTTP/1.1
</span><span class="stx-line" id="code_93">   is used as the underlying protocol, full duplex communication may be
</span><span class="stx-line" id="code_94">   broken if the client, server or any proxy chooses to buffer or reject
</span><span class="stx-line" id="code_95">   earlier 2xx responses.  Since proxies may buffer response body,
</span><span class="stx-line" id="code_96">   communication over WiSH may experience extra latency compared to
</span><span class="stx-line" id="code_97">   WebSocket.  When HTTPS is used, response buffering by proxies is
</span><span class="stx-line" id="code_98">   unlikely an issue.
</span><span class="stx-line" id="code_99">
</span><span class="stx-line" id="code_100">   Features dedicated for WebSockets which can be substituted by lower
</span><span class="stx-line" id="code_101">   layer protocols are dropped.  The WiSH protocol doesn't perform the
</span><span class="stx-line" id="code_102">   WebSocket handshake.  The fragmentation is only for enabling starting
</span><span class="stx-line" id="code_103">   message transmission before determining the final length of the
</span><span class="stx-line" id="code_104">   message.  No control messages are inserted between data messages as
</span><span class="stx-line" id="code_105">   control signals are exchanged in the lower layer protocols.
</span><span class="stx-line" id="code_106">   Relevance of the WebSocket status code and reason is to be studied.
</span><span class="stx-line" id="code_107">
</span><span class="stx-line" id="code_108">
</span><span class="stx-line" id="code_109">
</span><span class="stx-line" id="code_110">
</span><span class="stx-line" id="code_111">
</span><span class="stx-line" id="code_112">Yoshino &amp; Zhu           Expires January 26, 2017                [Page 2]
</span><span class="stx-line" id="code_113">
</span><span class="stx-line" id="code_114">Internet-Draft                WiSH Protocol                    July 2016
</span><span class="stx-line" id="code_115">
</span><span class="stx-line" id="code_116">
</span><span class="stx-line" id="code_117">   Application-level protocols may use WiSH as the framing protocol to
</span><span class="stx-line" id="code_118">   support bi-directional communication over HTTP and for Web and
</span><span class="stx-line" id="code_119">   Internet (mobile native) clients.
</span><span class="stx-line" id="code_120">
</span><span class="stx-line" id="code_121">2.  Conformance Requirements and Terminology
</span><span class="stx-line" id="code_122">
</span><span class="stx-line" id="code_123">   The key words &quot;MUST&quot;, &quot;MUST NOT&quot;, &quot;REQUIRED&quot;, &quot;SHALL&quot;, &quot;SHALL NOT&quot;,
</span><span class="stx-line" id="code_124">   &quot;SHOULD&quot;, &quot;SHOULD NOT&quot;, &quot;RECOMMENDED&quot;, &quot;MAY&quot;, and &quot;OPTIONAL&quot; in this
</span><span class="stx-line" id="code_125">   document are to be interpreted as described in [RFC2119].
</span><span class="stx-line" id="code_126">
</span><span class="stx-line" id="code_127">   Requirements phrased in the imperative as part of algorithms (such as
</span><span class="stx-line" id="code_128">   &quot;strip any leading space characters&quot; or &quot;return false and abort these
</span><span class="stx-line" id="code_129">   steps&quot;) are to be interpreted with the meaning of the key word
</span><span class="stx-line" id="code_130">   (&quot;MUST&quot;, &quot;SHOULD&quot;, &quot;MAY&quot;, etc.) used in introducing the algorithm.
</span><span class="stx-line" id="code_131">
</span><span class="stx-line" id="code_132">   Conformance requirements phrased as algorithms or specific steps can
</span><span class="stx-line" id="code_133">   be implemented in any manner, so long as the end result is
</span><span class="stx-line" id="code_134">   equivalent.  In particular, the algorithms defined in this
</span><span class="stx-line" id="code_135">   specification are intended to be easy to understand and are not
</span><span class="stx-line" id="code_136">   intended to be performant.
</span><span class="stx-line" id="code_137">
</span><span class="stx-line" id="code_138">3.  WiSH Protocol
</span><span class="stx-line" id="code_139">
</span><span class="stx-line" id="code_140">   The WiSH protocol transports messages over the message body of the
</span><span class="stx-line" id="code_141">   HTTP request message and the HTTP response message using the framing
</span><span class="stx-line" id="code_142">   defined in Section 4.
</span><span class="stx-line" id="code_143">
</span><span class="stx-line" id="code_144">   The Content-Type header value of the underlying HTTP request/response
</span><span class="stx-line" id="code_145">   message MUST be &quot;application/x-wish&quot;.
</span><span class="stx-line" id="code_146">
</span><span class="stx-line" id="code_147">4.  Framing
</span><span class="stx-line" id="code_148">
</span><span class="stx-line" id="code_149">   The WiSH protocol uses the framing defined in [RFC6455] for the
</span><span class="stx-line" id="code_150">   WebSocket protocol.
</span><span class="stx-line" id="code_151">
</span><span class="stx-line" id="code_152">   The opcode field indicates how to interpret the payload data field.
</span><span class="stx-line" id="code_153">   The WiSH protocol uses the following opcodes.
</span><span class="stx-line" id="code_154">
</span><span class="stx-line" id="code_155">   o  %x0 denotes a continuation frame
</span><span class="stx-line" id="code_156">
</span><span class="stx-line" id="code_157">   o  %x1 denotes a text frame
</span><span class="stx-line" id="code_158">
</span><span class="stx-line" id="code_159">   o  %x2 denotes a binary frame
</span><span class="stx-line" id="code_160">
</span><span class="stx-line" id="code_161">   Any value not listed here are reserved.
</span><span class="stx-line" id="code_162">
</span><span class="stx-line" id="code_163">
</span><span class="stx-line" id="code_164">
</span><span class="stx-line" id="code_165">
</span><span class="stx-line" id="code_166">
</span><span class="stx-line" id="code_167">
</span><span class="stx-line" id="code_168">Yoshino &amp; Zhu           Expires January 26, 2017                [Page 3]
</span><span class="stx-line" id="code_169">
</span><span class="stx-line" id="code_170">Internet-Draft                WiSH Protocol                    July 2016
</span><span class="stx-line" id="code_171">
</span><span class="stx-line" id="code_172">
</span><span class="stx-line" id="code_173">   The FIN bit together with the continuation frame opcode, payload len
</span><span class="stx-line" id="code_174">   and extended payload length work in the same way as the WebSocket
</span><span class="stx-line" id="code_175">   protocol to represent frames and messages.
</span><span class="stx-line" id="code_176">
</span><span class="stx-line" id="code_177">   RSV1, RSV2 and RSV3 are set to 0.
</span><span class="stx-line" id="code_178">
</span><span class="stx-line" id="code_179">   The MASK bit is always set to 0 in both server to client and client
</span><span class="stx-line" id="code_180">   to server direction as masking is not needed.  Therefore, the
</span><span class="stx-line" id="code_181">   masking-key field is never present.
